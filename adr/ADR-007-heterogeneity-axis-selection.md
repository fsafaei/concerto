## Context
v0.2 ¬ß3.2 requires each axis to produce a ‚â•20pp homogeneous-vs-heterogeneous gap or be cut. ADR locks which axes survive Phase-0 spike testing. The decision is bounded by three parallel inputs: the HMARL literature (which identifies axes with empirical precedent), the DACH customer-discovery cohort (which identifies axes with commercial weight in the German-speaking integrator market), and ‚Äî added in revision 2 ‚Äî an international (non-DACH) industry / applied-literature evidence sweep covering manufacturing & assembly, logistics & warehousing, and medical / construction / mining / defence verticals (see `adrs/international_axis_evidence.md`, 2026-05-08). The HMARL and international sources can be resolved now; the DACH synthesis and Phase-0 axis spikes remain pending.

## Decision
Adopt **Option D** ‚Äî the 6-axis shortlist (control rate, action space, observation modality, communication, partner familiarity, safety), implemented **progressively in three Phase-0 stages** (Stage 1: action space + observation modality; Stage 2: control rate + communication; Stage 3: partner familiarity + safety) with stage-to-stage progression gated on at least one ‚â•20 pp pass per stage. Each axis is provisionally retained pending its own ‚â•20 pp gap demonstration; axes that fail are cut without re-derivation. Staging order is set by implementation cost, dependency on other ADRs, and the fail-fast information value of each axis (see "Implementation staging" section). "Control-loop hierarchy / chunk-async coupling" is held for a separate post-spike review (see Open questions) rather than included as a 7th pre-registered axis. Empirical per-axis gap measurement (the ‚â•20 pp gate) is the ¬ßValidation criteria gate for promoting this ADR to Validated; until the spikes run, the surviving-axis set is data-dependent and downstream ADRs (008, 014) carry the corresponding Open work flags.

## Implementation staging
The 6 axes are not all equally cheap to instrument, and they do not all carry the same information value at the same time. The progressive staging below orders the spikes so that (a) the test rig is validated before expensive axes are touched, (b) the project's #verify priority (control rate) and the axis with sharpest applied numeric priors (communication) are resolved before partner-zoo or safety-filter dependencies are taken on, and (c) the most expensive spike (safety, possibly HIL) runs last when the supporting ADRs (-004, -006, -009) are mature enough to inform the test design. Each stage is gated: progression to stage N+1 requires ‚â•1 axis from stage N to clear the ‚â•20pp threshold (failure of *all* axes in a stage triggers a rig / methodology review before further compute is committed).

### Stage 0 ‚Äî Rig validation (precondition)
ADR-001's 50-line acceptance test (instantiate a 3-robot heterogeneous environment in ManiSkill v3, apply the per-agent action-repeat wrapper and the comm-shaping wrapper, run 100 control steps; pass criteria per ADR-001) must pass before any Stage-1 spike is run. This is not an axis spike ‚Äî it is the precondition that the heterogeneous test bench exists at all. ADR-007 spike work cannot start until ADR-001's smoke test is green.

### Stage 1 ‚Äî Foundation axes: action space + observation modality
- **Why first.** Cheapest to instrument (both available in ManiSkill v3 out of the box: MultiAgent action-space mixing for AS, RGB-D + state observation switching for OM). Strongest empirical priors (AS is the canonical HetGPPO / HARL axis; OM is the project's most-studied axis in industrial peg-in-hole literature). Failure here means the rig is broken, not the axis ‚Äî useful diagnostic signal. No dependency on other ADRs beyond the ADR-001 smoke test.
- **AS spike.** 7-DOF arm vs 2-DOF differential-drive base on a shared pick-place task; metric = episode success rate over 100 episodes; condition = MAPPO shared-parameter baseline vs per-agent parameters per HARL Fig. 8.
- **OM spike.** Same arm + base pair; condition = vision-only observation vs vision + force-torque + proprioceptive state fused; metric = same.
- **Stage 1 gate.** Proceed to Stage 2 if AS *or* OM shows ‚â•20pp gap. If both fail, pause: the ManiSkill v3 wrappers, the partner pairing, or the task selection is wrong ‚Äî this is rig debug, not axis cut.

Revision 4 (2026-05-15) splits Stage 1 into Stage 1a (rig-validation, ships now on the MPE stand-in) and Stage 1b (real-env science evaluation, deferred to Phase 1). The split exists because the Phase-0 chamber-side Stage-1 adapter ships on `MPECooperativePushEnv` as a CPU-friendly stand-in (no SAPIEN dependency on the maintainer's dev box) ‚Äî useful for rig validation but unable to evaluate the AS/OM axes as ¬ßValidation criteria defines them. The science-validity work is sequenced into Stage 1b without changing the ¬ßDecision: both sub-stages still target the AS + OM axes, the ‚â•20 pp gate still binds, and the per-condition pre-registrations are unchanged.

### Stage 1a ‚Äî Rig validation (Phase 0, MPE stand-in)
- **What runs.** `chamber.benchmarks.stage1_as.run_axis` and `chamber.benchmarks.stage1_om.run_axis` against `chamber.envs.mpe_cooperative_push.MPECooperativePushEnv`. Production ego is `_zero_ego_action` (always-zero action vector) by design ‚Äî Stage 1a's purpose is to exercise the rig (env steps, partner acts, episode terminations, success thresholding, SpikeRun aggregation, ADR-008 evaluation pipeline) so that any wiring regression trips before Stage 1b launches; a learning ego would confound the rig signal with the trainer signal.
- **What it does NOT do.** Stage 1a's MPE stand-in cannot evaluate the AS axis as ¬ßValidation criteria defines it (the 7-DOF panda vs 7-DOF panda + 2-DOF fetch-base contrast lives in a real manipulation env, not in 2-DOF MPE). The ‚â•20 pp gate is **not** measured under Stage 1a; the chamber-spike CLI surface, the pre-registration verification flow (ADR-007 ¬ßDiscipline), and the partner-stack dispatch are.
- **Stage 1a gate.** None ‚Äî Stage 1a is rig-validation, not axis evaluation. Failure here is debugged before Stage 1b is launched; success here is a precondition for cutting the Stage-1b pre-registration tags.

### Stage 1b ‚Äî Real-env science evaluation (Phase 1; trigger guardrail below)
- **What runs.** `chamber.envs.stage1_pickplace` (a thin ManiSkill v3 wrapper over the canonical pick-place task with the panda + fetch tuple) under both AS conditions, and the same pair under both OM observation modalities. The ego is trained per seed via `concerto.training.ego_aht.train` for the heterogeneous condition; the homogeneous condition uses the shared-parameter MAPPO baseline per the existing pre-registration. The pre-registrations issued for Stage 1 (`spikes/preregistration/{AS,OM}.yaml`) carry forward unchanged ‚Äî Stage 1b uses the same condition IDs, seed list, and episode budget; only the env implementation behind the condition IDs changes.
- **Compute budget.** 5 seeds √ó 100k training frames per axis = 1M frames per axis on A100 (‚âà5 GPU-h per axis); the pacluster bootstrap (ADR-008 ¬ßDecision) runs over the 5 per-seed trained policies √ó 20 evaluation episodes per (seed, condition).
- **Stage 1b gate.** ‚â•20 pp gap in either AS or OM (the ¬ßValidation criteria gate). Clearance promotes ADR-007 to **Validated** for those axes and unlocks Stage 2.
- **Trigger guardrail.** Stage 1b launches as the first Phase-1 milestone, **before any Stage-2 spike is permitted, no later than 4 weeks after the Month-3 lock review**. This pin exists so the science-validity deferral does not become permanent: Stage 2 (CR + CM) cannot draw on a Stage-1 ‚â•20 pp signal that has never been measured, and the Month-3 lock review's "Stage-1 rig-validation cleared" status only stays defensible if Stage-1b lands inside the four-week window.

### Stage 1b precondition for Stage 1a (operational)
- The Stage 1b pre-registration git tags (`prereg-stage1-AS-‚Ä¶`, `prereg-stage1-OM-‚Ä¶`) MUST be cut on the same on-disk YAML blob SHAs the Stage 1a runs verified ‚Äî re-using the canonical pre-registrations is the entire point of the sub-stage split. ### Stage 1b implementation details (Revision 5, 2026-05-17; P1.03)

The Stage-1b env (`chamber.envs.stage1_pickplace.Stage1PickPlaceEnv`, P1.03) lands with three implementation choices that are **not** comparison-protocol changes (so the locked pre-registration YAMLs / tags stay untouched per ¬ßDiscipline), but that downstream reviewers must be able to find without grepping source. They are recorded here so the audit trail is closed.

1. **Joint-space action / Cartesian-CBF approximation (Q3 mod 1).** Stage-1b uses `pd_joint_delta_pos` for the panda (7-DOF joint-position deltas) so the AS axis preserves its intended "7-DOF arm vs 2-DOF differential-drive base" contrast (collapsing to `pd_ee_delta_pose` would reduce the panda's action to 6-D Cartesian and gut the AS axis's heterogeneity). `JacobianControlModel.action_to_cartesian_accel` projects through `J(q) @ action_position_delta`, which is dimensionally a Cartesian *position* (m), not a Cartesian *acceleration* (m/s¬≤) ‚Äî off by `1/dt¬≤` (~2500√ó at `dt=0.02s`, small-Œîp scale) from the true `ddot p` the SAPIEN PD controller produces after `œÑ ‚Üí physics ‚Üí next state`. The conformal-slack overlay (Huriot & Sibai 2025 ¬ßVI) is designed for exactly this class of constraint-RHS modelling error: the per-pair slack Œª converges to a non-trivial steady-state value reflecting the (Œîp ‚Üî ddot p) scale mismatch. The Stage-2 CR-axis review (Stage 2 sub-section above) re-examines whether the control-mode / control-rate interaction admits a tighter analytical mapping; the structural fix is sequenced as slice **P1.05.5** below.

2. **Œª-telemetry handoff contract (Q3 mod 2).** Option-1's whole credibility rests on the conformal overlay absorbing the dynamics-mismatch error empirically, not just in theory. The Stage-1b archive MUST carry per-`(seed, condition)` Œª trajectories so the post-run audit can confirm (a) Œª is actually adapting (not pinned at 0); (b) Œª does not saturate against `Bounds.cartesian_accel_capacity` (saturation would mean the QP is effectively running without safety margin); (c) steady-state Œª is finite and consistent across seeds. ### Stage 2 ‚Äî Empirical-uncertainty axes: control rate + communication
- **Why second.** Both axes need a custom wrapper layered on the validated Stage-1 rig: control rate needs two independently-clocked control loops (the action-repeat wrapper from ADR-001 generalised to per-agent rates), communication needs a degradation wrapper on the fixed-format channel (ADR-003). Both have specific isolation requirements that benefit from running after the rig is validated. CR is the project's #verify priority (HetGPPO ¬ß3.4 + HARL ¬ß3.4 both flag it absent from HMARL literature). CM has the sharpest numeric priors of any axis (3GPP Release 17 + URLLC + arXiv 2501.12792). Independent of each other ‚Äî can run in parallel if compute permits.
- **CR spike.** Single-embodiment robot (e.g. one 7-DOF arm) with two independently-clocked control loops at 500 Hz vs 50 Hz, chunk size held constant, action-space dimensionality held constant ‚Äî the deliberate single-variable design that mitigates Risk #2. Additionally instrumented with a 5‚Äì15 Hz partner-VLA condition (chunk size = 1 vs 8 vs 50 steps) so the chunk-async sub-question (Open question #3) is answered as a side-effect.
- **CM spike.** Latency / jitter / drop sweep on the fixed-format channel using the URLLC and 5G-TSN industrial-trial numbers as anchor points (latency 1‚Äì100 ms, jitter Œºs‚Äì10 ms, drop 10‚Åª‚Å∂ to 10‚Åª¬≤) per arXiv 2501.12792 and 3GPP Release 17. Fixed-format channel only ‚Äî learned-message overlay (ADR-003) is held for Phase 1.
- **Stage 2 gate.** Proceed to Stage 3 if CR *or* CM shows ‚â•20pp gap. If both fail, the project's heterogeneity claim weakens substantially: ¬ß3.4 needs revision, ADR-009 (partner zoo) and ADR-011 (baseline set) cannot lock until ¬ß3.4 is re-derived, and the Month-3 senior-advisor review must re-examine whether the project's positioning against Liu 2024 RSS / COHERENT / Huriot‚ÄìSibai 2025 still holds at 2 axes (AS + OM).

### Stage 3 ‚Äî Systems-dependent axes: partner familiarity + safety
- **Why last.** Both axes depend on other ADRs being mature. PF requires at least an embryonic ADR-009 partner zoo (a 3-partner draft is enough: heuristic, frozen MAPPO checkpoint, frozen RL checkpoint). SA requires ADR-004's exp CBF + conformal overlay + OSCBF stack functional, ADR-006's per-task numeric bounds enumerated, and a contact-rich task setup that exercises the per-vendor force-limit / SIL-PL variance under ISO 10218-2:2025; sim may be insufficient and an HIL spike may be needed. Running these last lets the supporting ADRs catch up; running them earlier risks invalidating spikes when the supporting infrastructure changes.
- **PF spike.** Trained-with vs frozen-novel partner from a 3-partner ADR-009 draft zoo; metric = episode success rate; manipulation = swap the partner mid-episode in the frozen-novel condition to test the conformal slack Œª re-init (ADR-004 / ADR-006 dependency).
- **SA spike.** Heterogeneous force-limit / SIL-PL pair on a contact-rich scenario (handover or co-carry); metric = task success √ó (1 ‚àí violation rate) per ADR-014's three-table format. HIL escalation if the simulation cannot capture per-vendor controller-stack variance under ISO 10218-2:2025 ‚Äî this is the single most expensive spike in Phase 0 and should be sequenced last in any case.
- **Stage 3 gate.** Each axis is cut individually if it fails ‚â•20pp. PF failure is recoverable ‚Äî it weakens but does not invalidate the project's AHT claim. SA failure is more consequential: it forces an ADR-004 + ¬ß6.2 review to determine whether the safety axis decomposes (Open question #4) or is dropped from ¬ß3.4 entirely. SA failure does *not* invalidate the project's safety claim ‚Äî that is a property of the conformal CBF stack, not of safety-as-¬ß3.4-axis.

## Rationale
The published HMARL literature narrows the candidate set to axes with empirical precedent, the DACH customer-discovery playbook provides an independent commercial ranking that converges on the same axes, and the international industry / applied literature corroborates and tightens that ranking with deployment-grade signal. HetGPPO (notes/tier1/23_hetgppo.md ¬ß2, Fig. 1) establishes the P √ó B_s √ó B_d taxonomy with five mutually exclusive heterogeneity subclasses, giving direct empirical grounding for observation-space and action-space axes; both HetGPPO (¬ß3.4 #verify) and HARL (notes/tier1/25_harl.md ¬ß3.4) flag control-rate heterogeneity as absent from the HMARL literature but structurally present in the target platform (manipulator at ~500 Hz vs. mobile base at ~50 Hz). The DACH discovery playbook (discovery_dach_playbook.docx ¬ß2 secondary hypotheses) independently names control rate, observation modality, and action space as the axes to probe first, and explicitly tests whether communication degradation (latency, drop, jitter) is a top-3 integrator pain ‚Äî aligning with the literature shortlist; the conversation-synthesis template (¬ß8.1) operationalises a 6-axis heterogeneity ranking ‚Äî control rate / action space / observation modality / communication / partner familiarity / **safety** ‚Äî extending Option B by one axis. International evidence (`adrs/international_axis_evidence.md` ¬ß2, 2026-05-08) corroborates each of those six axes with non-DACH signal: (i) control-rate heterogeneity is now load-bearing in the post-OpenVLA wave ‚Äî Real-Time Chunking [arXiv 2509.23224], A2C2 [arXiv 2512.20188], FAVLA [arXiv 2602.23648], Asynchronous Fast-Slow VLA [arXiv 2512.20188], VLSA / AEGIS [arXiv 2512.11891], and SilentDrift [arXiv 2601.14323] all treat slow-VLA / fast-controller mismatch as the dominant failure mode, and every confirmed 2025 humanoid factory deployment (BMW √ó Figure 02 at Spartanburg, Mercedes √ó Apptronik Apollo, Toyota √ó Atlas, BYD √ó UBTECH, Foxconn √ó Robust.AI) couples a slow chunked-action partner with kHz-grade industrial arms; (ii) action-space heterogeneity is supported by Open X-Embodiment [arXiv 2310.08864] embodiment-scaling laws, XMoP zero-shot transfer to seven commercial arms, and CrossFormer's "no-positive-transfer-from-co-training" replication; (iii) observation-modality heterogeneity is treated as load-bearing in the visual-tactile peg-in-hole assembly literature (MDPI Machines 2025; HIT 2025 patent ensembles) and in the multimodal-fusion + VLM survey for robot vision [arXiv 2504.02477]; (iv) communication degradation has tighter numerical priors than the v0.2 plan presumed ‚Äî 3GPP Release 17 standardises 5G-as-virtual-TSN-bridge with uplink time sync, URLLC delivers 1 ms latency at 99.9999% reliability, and factory-jitter measurements in arXiv 2501.12792 give per-task numeric bounds usable directly in ADR-006; (v) partner familiarity / black-box partner is supported by MALMM (IROS 2025), LLM-Coordination (NAACL 2025 findings), EMOS, the LLM-for-Heterogeneous-AHT line [arXiv 2406.12224], and the TMLR 2025 generalisation-to-novel-partners study (Ruhdorfer et al.); (vi) safety heterogeneity benefits from a regulatory-window argument that did not exist when v0.2 was drafted ‚Äî ISO/TS 15066 has been integrated into ISO 10218-2:2025, replacing the 2011 versions, so mixed-vendor cells are now compliance-heterogeneous in a binding-standard sense; surgical multi-arm deployments (Versius / Hugo / da Vinci) report different arm-collision rates across vendors, and mining (Caterpillar Cat Command running on Komatsu 930E trucks) is the canonical commercial example of one safety stack spanning multi-vendor controllers. Taken together, the DACH ¬ß9 go/no-go rule, applied to the international cohort, would PROCEED-WITHOUT-ADJUSTMENTS by a wide margin; the DACH synthesis remains the contractually-committed gate but the international evidence reduces the prior probability that DACH will overturn Option D. - Other ADRs: ADR-009 (partner-zoo construction) and ADR-011 (baseline set) cannot be locked until this ADR is decided; ADR-008 (HRS bundle) requires the surviving axis set to parameterise the robot-capability descriptor; ADR-004 (safety filter formulation) is affected if safety heterogeneity is confirmed as a distinct axis; ADR-006 (partner-policy assumption set) can borrow per-task numeric latency / jitter / drop bounds directly from the 5G-TSN industrial-trial literature surfaced by the international sweep (URLLC 1 ms / 99.9999%, factory jitter measurements per arXiv 2501.12792), tightening the "explicit numeric bounds per benchmark task" gap noted in ADR-006.

## Risks and mitigations
- [axis fails gap test] One or more candidate axis fails the ‚â•20pp threshold, reducing the axis count below the plan's diversity target ‚Üí mitigation: axes that fail are cut with a single ¬ß3.4 revision before Phase 1 locks; spike baseline pair (homogeneous vs. heterogeneous) must be pre-registered before running to prevent retroactive threshold adjustment; the international evidence in adrs/international_axis_evidence.md serves as a *prior* on which axes are most likely to pass and is **not** a substitute for the empirical gap measurement.
- [control-rate confound] Control-rate heterogeneity may be inseparable from action-space heterogeneity in the spike setup (arm at 500 Hz also has a different action space from base at 50 Hz) ‚Üí mitigation: design the spike so the only manipulated variable is update frequency; use a single-embodiment robot with two independently-clocked control loops to isolate the effect before claiming control-rate as a standalone axis. International evidence (RTC, A2C2, Asynchronous Fast-Slow VLA) sharpens this risk: it shows that "control rate" and "action chunk length" interact in deployed VLA partners (the SilentDrift result that 1 mm/step compounds to 5 cm over 50 chunks), so the spike design must record both per-step action delta and chunk size to distinguish the two effects.
- [discovery adds safety axis, expands scope] If the 8-week cohort synthesis returns PROCEED-WITH-ADJUSTMENTS with safety as a new axis, ADR-004 (safety filter) and ¬ß6.2 must be revisited to handle per-vendor heterogeneous force limits ‚Äî a non-trivial scope expansion ‚Üí mitigation: treat safety as a working candidate axis now (matching discovery_dach_playbook.docx ¬ß8.1 *and* adrs/international_axis_evidence.md ¬ß2.6) and run the safety-axis spike in parallel with the other 5; if the spike fails the ‚â•20pp test, it is cut regardless of discovery signal.
- [discovery contradicts literature shortlist] DACH integrators may rank communication and partner-familiarity far above control-rate or obs-modality, forcing a reweight that de-prioritises the literature's most-evidenced axes ‚Üí mitigation: the ¬ß9 decision rule is pre-committed; any reweight must come from the final synthesis applying the rule, not from informal impressions during fieldwork; the international evidence sweep is a cross-check ‚Äî if DACH alone reweights but the international applied literature does not, the reweight should be flagged as a DACH-local effect at Month-3 review and not silently propagated.
- [ISO 10218-2:2025 regulatory shift] ISO/TS 15066 has been absorbed into the binding ISO 10218-2:2025 standard since v0.2 was drafted, so mixed-vendor cells are now subject to a single binding standard rather than a technical specification ‚Üí mitigation: treat the safety axis specifically as compliance-heterogeneous-controller-stack heterogeneity (per-vendor force-limit pre-commits *and* per-vendor functional-safety SIL/PL ratings), not just as ISO/TS 15066 force-limit variance; ADR-004 (safety filter) and ADR-014 (safety reporting) must explicitly reference 10218-2:2025 when the safety axis is locked. See Open question #4 for the decomposition question.
- [chunk-async coupling deserves its own axis] The post-OpenVLA cluster (RTC, A2C2, Fast-Slow VLA, FAVLA) treats two-policy-decoupled-clocks-with-chunk-inpainting as a distinct engineering problem rather than a special case of control-rate heterogeneity ‚Üí mitigation: do **not** add a 7th pre-registered axis without spike evidence (the working recommendation explicitly excludes this); instead, instrument the control-rate spike to record whether the gap survives when chunk size is held constant (i.e., is the gap from rate alone, or from rate √ó chunk-length interaction?). Resolution moves to Open question #3 post-spike.

## Reversibility
Type-2 (two-way door): the axis list can be revised during Phase 0 as spike results arrive and the discovery synthesis completes. However, any axis included in the Decision that drives partner-zoo construction (ADR-009) or benchmark task selection (ADR-001, ADR-005) becomes effectively Type-1 once those downstream ADRs are accepted ‚Äî exit ramp at that point requires a new ADR superseding ADR-007 plus cascading updates to ADR-001, ADR-005, ADR-009, and ADR-011. Adding a 7th axis (chunk-async coupling) post-spike is Type-2 if done before ADR-009 locks and Type-1 thereafter; the schedule should reflect this constraint (axis list must finalise before partner-zoo construction begins).

## Validation criteria
By Phase-0 gate (Month 3): each axis selected in the Decision produces a ‚â•20 percentage-point gap in task-success rate between homogeneous and heterogeneous agent pairs on at least one benchmark scenario; metric = episode success rate averaged over 100 evaluation episodes; homogeneous baseline = shared-parameter MAPPO; heterogeneous condition = same task with the axis-specific manipulation applied. Per-axis manipulation:
- *Control rate.* Two independently-clocked control loops on a single embodiment, e.g. 500 Hz vs 50 Hz, with chunk size held constant; 5‚Äì15 Hz partner-VLA condition added to instrument the chunk-async sub-question (Open question #3).
- *Action space.* Two embodiment classes with different action-space dimensionality (e.g. 7-DOF arm vs 2-DOF differential-drive base) sharing one task.
- *Observation modality.* One condition with vision-only observation, one with vision + force-torque + tactile fused.
- *Communication.* Latency / jitter / drop sweeps anchored to the URLLC and 5G-TSN industrial-trial numbers (latency 1‚Äì100 ms; jitter Œºs‚Äì10 ms; drop 10‚Åª‚Å∂ to 10‚Åª¬≤) per arXiv 2501.12792 and 3GPP Release 17.
- *Partner familiarity.* Trained-with vs frozen-novel partner from the partner zoo (ADR-009).
- *Safety.* Heterogeneous force-limit / SIL/PL pair (matching ISO 10218-2:2025 compliance variance) on at least one contact-rich scenario; HIL spike if simulation cannot capture the per-vendor controller-stack variance.

In addition to the empirical ‚â•20pp gap, at least one DACH integrator conversation (per discovery_dach_playbook.docx ¬ß8.1 synthesis) independently names the axis as a real production pain, providing commercial corroboration; the international evidence sweep (adrs/international_axis_evidence.md ¬ß3) supplies a non-DACH commercial cross-check but is not part of the formal go/no-go rule.
